bench-wrapper.sh：零成本轨迹追踪

框架不硬编码编译命令——bench-wrapper.sh 是模板，agent 将实际 bench 命令填入 {{BENCH_COMMAND}} 生成 scripts/bench.sh。每次执行自动将 solution/ 快照到 trajectory/，附上 output.txt。

自动 timestamp 代码快照 输出归档 失败也记录 
# 每次 bench 自动存档

mkdir -p "$TRAJ_DIR"
cp -r solution/* "$TRAJ_DIR/"   # 代码快照
mv _bench_output.txt \
   "$TRAJ_DIR/output.txt"       # 输出归档

Section 03

跨实验核心规律

💡
以下规律从 16 个实验案例的 ITERATIONS.md 中人工提炼，代表 agent 在不同算子上多次独立发现的稳定结论。
✅ 通用有效优化
  • CFG_MDL 总优于 CFG_NORM（dsa Iter13, matmul Iter13）— MDL 大包 MTE2 对矩阵类算子几乎是默认最优，无需反复验证
  • enUnitFlag 几乎总是有益（dsa Iter5 +15%, matmul-asc-v5）— Cube∥FixPipe 并行，~10-20% 免费提升
  • 多核 N-split 是 MTE2 瓶颈突破口（dsa Iter9, matmul Iter51, matmul-py Iter79）— 并行带宽 > 单核优化上限，是最大单步提升来源
  • Atlas 910B 最优核数 8-16（多案例一致）— 更多核 Scalar overhead 超越收益，常见于 16 核是甜点
  • enableStaticPadZeros + enableInit=false（dsa Iter18）— 对固定 shape 额外减少 ~5% Scalar 开销
❌ 常见陷阱与误判
  • MultiCoreMatmulTiling 的 singleCoreM bug — Iterate 忽略该参数，每核重复处理全量 M，导致精度失败，必须改用 MatmulApiTiling 手动偏移
  • MTE2 preload 对小 shape 有害（dsa Iter7, matmul Iter9）— 预加载开销 > 收益，仅对大 shape 迭代次数多时有效
  • 常量 Tiling 在 Atlas A2 不稳定（matmul-asc-v5 Iter3: 2× 变慢！）— 与 GPU 经验相反，慎用
  • Vector kernel 不能替代 Cube 做 GEMM（dsa Iter25: 1.4s vs 4µs）— 算力差 ~100×，矩阵操作必须用 Cube API
  • Bench 方法论本身是大坑（attn-bwd 前 14 轮废了）— 含 host I/O 的计时不代表 kernel 性能，必须先用 msprof 稳定测量
  • MIX mode（aicore + cube）在 CANN 8.3 受限（dsa Iter25: DDR 地址限制）— __aicore__ 无法访问部分 GM 地址区域

Section 04

复现环境要求

⚠️
无法在 GPU 或 CPU 上仿真复现 msprof、Ascend C 编译器（bisheng）、NPU 内核执行均需真实昇腾硬件。CANN 提供的仿真模式功能受限，无法复现性能数据。

硬件要求

  • 必
    华为昇腾 NPU 实验硬件：Atlas A2 / Ascend 910B。其他型号（950PR、A3）可能需调整部分 API（如 enableRelu 仅 950 支持）
  • 必
    x86_64 服务器 实验服务器：Intel Xeon Platinum 8468V（AVX-512，用于 host 端优化）
  • 建议
    独占 NPU 卡 用 ASCEND_RT_VISIBLE_DEVICES=<卡号> 隔离，避免多任务干扰 msprof 时序

软件版本

  • 必
    CANN 8.3.RC1 实验环境版本。部分 API（__kfc_workspace__、MIX mode 行为）版本间有差异
  • 必
    bisheng 编译器 LLVM-based，含 Ascend C 前端。注意：OpenMP 运行时为 __kmpc_*（LLVM），与 libgomp ABI 不兼容
  • 必
    msprof 性能工具 op level profiling，输出 OpBasicInfo / PipeUtilization / Memory CSV。--warm-up=10 --launch-count=10 为标准参数
  • 可选
    npu-smi HINTS.md 特别说明：无 npu-smi 不代表 NPU 不可用，以实际编译运行为准
  • i
    Claude Code CLI 用 claude --dangerously-skip-permissions 运行，.claude/settings.local.json 已预设全权限

Section 05

后续优化方向

以下三个层面的改进方向，基于对 16 个实验案例中 agent 行为的人工观察和问题归因。 每项均标注了实现难度和预期收益。

4.1 Agent 框架层

📊
停滞检测：从计数到统计显著性
★★★★★ 优先
当前问题：用「连续 3/5 轮无改善」的死计数规则。但 NPU 延迟有 ~10% 噪声，「真实改善」与「测量噪声」无法区分。实验中大量 no-change 迭代实为统计噪声，agent 却继续在错误方向打转。
  • 将 ITERATIONS.md 模板增加 Runtime(std) 字段，bench 脚本强制跑 5 次取均值
  • 停滞判定改为：效应量 < ε（如 3%）且 std_overlap > 50%，才算真正无改善
  • 加速比只比较 mean，不允许用 best-of-N 代替 mean 汇报
实现难度：低（改 Markdown 模板） 预期收益：减少无效迭代 ~30%
🗂
FINDINGS.md：结构化历史知识积累
★★★★★ 优先
当前问题：ITERATIONS.md 是自由文本，上下文压缩后 agent 无法高效检索历史发现。实验中 agent 在 100+ 迭代后频繁重复已尝试的配置（如多次尝试 MTE2 preload 已确认失败的参数组合）。
  • 新增 FINDINGS.md，分「已确认有效」「已确认无效」「待验证」三栏
  • 每次迭代后强制更新，agent 制定下一步时优先读 FINDINGS.md 而非翻历史迭代
  • 无效列表尤其重要：如「enableSetTail=false 精度失败」「32 核会 hang」「Vector kernel 替代 Cube 慢 100×」
实现难度：低（新增模板文件） 预期收益：减少重复尝试，节省 budget
🏗
分层探索：架构决策先于参数调优
★★★★☆ 次优先
当前问题：agent 把架构级决策（多核 vs 单核、算法路径）和参数调优（baseN=128 vs 256）混在一起。matmul-asc-v5 最典型：50 轮单核参数调优只获 1.18×，Iter 51 换 API 一步跳到 5.36×。
  • 在 OPTIMIZE.md 中强制两阶段：Phase 1（架构探索，每方案仅 1 次快速验证）→ Phase 2（参数 sweep，在确定架构后）
  • 架构探索清单：① 单核 vs 多核 ② 是否用 Cube API ③ MIX mode 可行性 ④ 数据类型（bf16/fp32）
  • 理论极限计算：在 Phase 1 结束前，根据 msprof 数据计算「当前架构的理论下界」，判断是否值得进入 Phase 2
实现难度：中（改 OPTIMIZE.md 流程） 预期收益：避免在错误架构上浪费 50+ 轮
⚡
E2E 视角自动引入时机
★★★★☆ 次优先
当前问题：dsa-v5 在 kernel 优化到 4.14 µs 后，花了近 100 轮才开始关注 E2E。当时 kernel 只占 E2E 的 3.3%，而 host TopK 占 84%。过度优化 kernel 是低效的。
  • HINTS.md 增加 E2E 触发条件：kernel 连续 5 轮无改善 AND 尚未测量 E2E 时间 → 强制插入 E2E 测量（clock_gettime 各阶段分项计时）
  • 若 kernel time < 20% E2E → 自动转向 E2E 最大瓶颈
  • 若「理论带宽利用率 > 70%」→ 可宣告 kernel 收敛，不再在 kernel 上投入
实现难度：低（改 HINTS.md） 预期收益：避免过度优化 kernel，及时发现真正瓶颈
🔀
并行实验流水线（多 agent sweep）
★★☆☆☆ 高收益高难度
当前问题：所有迭代串行执行，一次 msprof 就要 2-3 分钟，36 配置 sweep = 1-2 小时。200 迭代 = 整晚不停跑。
  • 利用 Claude Code 的多 agent 能力：主 agent 规划 K 个候选配置 → 派发 K 个子 agent 并行 bench → 汇总选最优
  • 前提条件：多 NPU 卡（ASCEND_RT_VISIBLE_DEVICES=0,1,2,3），或验证同卡多进程 msprof 并发限制
  • bench 脚本需改为无状态（输出到不同目录，不共享 trajectory/）
  • sweep 阶段效果最显著：36 配置从 2 小时压缩到 20-30 分钟
实现难度：高（需要多卡或并发支持验证） 预期收益：sweep 速度 5-10×

4.2 Auto-Research 方法论层

🗺
瓶颈→策略映射表内置到 OPTIMIZE.md
★★★★☆ 优先
当前问题：agent 的优化假设主要来自 skill 文档推荐，缺乏系统性。50+ 迭代后 agent 开始随机组合 flag，效率极低（dsa-v5 后期 80% 迭代是无效 flag sweep）。
  • 将「Section 03 瓶颈→策略对照表」直接内置到 OPTIMIZE.md，让 agent 每次 profiling 后强制对照查找
  • 增加「理论下界估算」模板：数据量 / 硬件峰值带宽 = 理论最小时间。current / theoretical < 1.3× → 该方向收敛，停止
  • 增加「搜索状态追踪」字段到 ITERATIONS.md：已穷举空间、待探索空间、预估剩余改善空间
实现难度：低（改 Markdown） 预期收益：减少后期无效 flag sweep

理论极限计算与自动收敛声明
★★★★☆ 次优先
当前问题：当前框架没有「宣告收敛」的机制，agent 会一直跑到 budget 用尽。dsa-v5 Iter 9 多核突破后，理论极限已经触及，但仍继续跑了 190+ 轮基本无效迭代（从 4.52→4.14 µs 仅 8.4%）。
  • 规则：任意停滞节点，agent 必须计算「数据移动下界」：
        理论最小时间 = Σ(数据量) / 硬件带宽峰值
        910B GM 带宽 ≈ 1.8 TB/s，per-core ≈ 1.8T/num_cores
  • 如果 current ≤ 1.5 × theoretical_min：提示「已接近硬件极限，当前方向收敛，转向 E2E 或其他维度」
  • HINTS.md 中给出各硬件资源参数（L0C、L1、UB 大小，GM 带宽），agent 可直接引用
实现难度：中（需要 skill 提供硬件参数） 预期收益：节省 30-50 轮无效迭代
