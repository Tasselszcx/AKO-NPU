# Task A 源码读取与溯源规则

对应 Task A 执行阶段 A/B/C/D：
- 步骤 1：读取 Scout 侦察报告，建立分支预期
- 步骤 2：读取 tiling P0 源码，提取分支骨架和 conditions
- 步骤 3：读取 tiling P1 源码（外部函数溯源），获取函数返回值
- 步骤 4：读取 kernel P0 dispatch 块，提取 key 映射表

---

## 首要步骤：读取 Scout 侦察报告

尝试读取 Phase 0 产出的 `S2P0_scout_t.md` 和 `S2P0_scout_k.md`（如存在）。

Scout 报告提供（辅助信息，不替代源码）：分支追踪路径及行号、tiling_key 公式、平台判断函数已确认返回值、kernel 逐条 dispatch 映射和总 key 计数、架构守卫求值结果。

仍需从源码提取（Scout 不提供）：结构化 conditions JSON、源码约束表逐字抄录、input_variables / caller_options / internal_variables 分类、kernel 内部分支的轴映射分析。

Scout 报告可用时的验证流程：
1. 从 Scout 获取分支层数、条件概要、key 公式、dispatch 总数建立预期
2. 按"源码读取范围"读取文件（与无报告时完全相同）
3. 对照验证：在源码行号确认条件表达式是否一致、分支是否遗漏、key 计数是否对得上
4. 不一致以源码为准，直接修正
5. 验证通过后可基于报告中的分支结构快速展开，无需从零推导拓扑

---

## 文件读取策略

严格按主 Agent 传入的"源码读取范围"文本块执行，未列出的文件禁止读取、排除列表中的文件禁止读取、不需自行读取 `S2P0_file_manifest.json`。`kernel.total_key_count` 是完整性校验基准。

---

## 发现缺失的外部符号

发现引用未在读取范围中列出且返回值影响分支判断的外部符号时，禁止自行读取其定义文件。在 `completeness_checklist` 中记录为 `missing`（evidence 说明符号名、调用位置、影响分支），由主 Agent 在返回后决定是否补充。

---

## 公共函数溯源验证

分析源码时，对当前文件未定义的函数调用必须溯源到实现代码，读取实际逻辑后再决定路径走向。

### 适用范围

需溯源：平台判断函数（返回值决定分支走向）、计算工具函数（返回值参与条件判断）、公共头文件中的 inline 函数。

不需溯源：标准库函数、框架基础设施（`context->GetInputShape()` / `SetBlockDim()` 等确定性接口）、已在当前文件有定义的函数。

### 强制流程

1. **识别**：发现函数调用在当前文件无定义
2. **定位**：通过 `#include` 链和目录结构找到实现文件
3. **读取**：读取实现代码，理解实际判断逻辑
4. **代入**：将目标平台参数值代入实现逻辑，得到确定返回值
5. **记录**：在 `source` 字段引用实现文件路径和行号（格式：`调用文件:行号 → 实现文件:行号`）

溯源深度：到函数实现代码并理解返回值逻辑即可，不需继续溯源更深层调用。

### 行为约束

禁止根据函数名/文件名/平台名推断行为；禁止跳过溯源直接假设函数行为；禁止读取"源码读取范围"之外的文件；发现缺失外部符号 → 在 `completeness_checklist` 报告为 missing。

遇到以下想法时必须 STOP，回归源码确认：
- "这个值大概是 N" → 找 `constexpr`/`#define` 确认精确值
- "这个算子应该跟那个类似" → 每个算子独立分析
- "这个约束太复杂了" → 不可跳过，写进未确认项
- "没找到路径" → 记录为"未发现"，不默认没有
- "这个平台应该走这个分支" → 找到分支条件实现，代入目标平台值验证
- "proto.h 说不支持" → 不做可达性判断，kernel 有实现就报告
- "这条路径太边缘了" → 所有路径都报告
- "这个分支跟主干差不多" → conditions 不同就是不同路径
- "这个函数名说明了它的行为" → 函数名≠实现，读代码

所有路径都报告，不做重要性判断；内部变量放入 internal_variables 但不可丢弃。
